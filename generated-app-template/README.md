# AI Services Marketplace

A real product slice generated by the Software Factory: a marketplace where
customers post AI service requests, an **AI brief** is generated for each
request, providers submit **proposals**, and customers **accept or reject**
them — every step persisted to a database and shown on the customer, provider,
and admin dashboards.

This app is **self-contained**: it has its own `package.json`, tooling, and
`node_modules`, and is verified by its own commands. It does **not** depend on
any factory packages.

- Stack: Next.js 15 (App Router) · React 19 · TypeScript (strict, ESM) · Prisma
- Local/test database: SQLite (no external services)
- Hosted database: Postgres on Render (documented below)
- Tests: Vitest (unit) · Playwright (e2e smoke)

## Quick start

```bash
pnpm install          # installs deps and runs `prisma generate` (postinstall)
cp .env.example .env  # sets DATABASE_URL="file:./dev.db"
pnpm db:setup         # prisma generate + db push + seed (creates prisma/dev.db)
pnpm dev              # http://localhost:3000
```

Open `/customer` to submit a request, `/provider` to propose, `/admin` for
health and activity.

## Environment variables

| Variable            | Required | Purpose                                                              |
| ------------------- | -------- | ------------------------------------------------------------------- |
| `DATABASE_URL`      | yes      | Prisma datasource. SQLite locally (`file:./dev.db`), Postgres on Render. |
| `AI_BRIEF_PROVIDER` | no       | When set with the API key, selects a live brief provider.           |
| `AI_BRIEF_API_KEY`  | no       | API key for the live brief provider.                                |

When `AI_BRIEF_PROVIDER` and `AI_BRIEF_API_KEY` are **not** set, brief
generation uses a deterministic, template-based fallback (used by tests/CI). The
`generateBrief` function is async so a real provider call can be slotted in
behind the same signature without changing any caller.

## Data model

Six entities (see `prisma/schema.prisma`):

| Entity           | Key fields                                                      | Relations                                  |
| ---------------- | -------------------------------------------------------------- | ------------------------------------------ |
| `Customer`       | name, email (unique)                                            | has many `ServiceRequest`                  |
| `Provider`       | name, email (unique), expertise                                 | has many `Proposal`                        |
| `ServiceRequest` | title, description, category, budget, status                   | belongs to `Customer`; has one `AIBrief`; has many `Proposal`, `StatusEvent` |
| `AIBrief`        | summary, scope, successCriteria, suggestedBudget, source       | belongs to one `ServiceRequest` (unique)   |
| `Proposal`       | message, price, status                                          | belongs to `ServiceRequest` + `Provider`; has many `StatusEvent` |
| `StatusEvent`    | type, message, createdAt                                        | belongs to `ServiceRequest`; optional `Proposal` |

Status / type values are stored as `String` (SQLite has no native enums):

- `ServiceRequest.status`: `OPEN` → `PROPOSED` → `ACCEPTED` (or `CANCELLED`)
- `Proposal.status`: `SUBMITTED` → `ACCEPTED` | `REJECTED`
- `StatusEvent.type`: `REQUEST_CREATED`, `BRIEF_GENERATED`, `PROPOSAL_SUBMITTED`,
  `PROPOSAL_ACCEPTED`, `PROPOSAL_REJECTED`

All field types are portable (`String` / `Int` / `DateTime` / relations) so the
same schema targets both SQLite and Postgres.

## API surface

All routes are under `src/app/api`. Mutating routes generate the appropriate
status events.

| Method  | Route            | Body / query                                                              | Response                              |
| ------- | ---------------- | ------------------------------------------------------------------------ | ------------------------------------- |
| `GET`   | `/api/requests`  | —                                                                        | `{ requests: ServiceRequest[] }` (with brief, proposals, status, customer) |
| `POST`  | `/api/requests`  | `{ customerName, customerEmail, title, description, category, budget }`   | `201 { request }` (brief + 2 status events created) |
| `GET`   | `/api/proposals` | `?requestId=` (optional)                                                  | `{ proposals: Proposal[] }`           |
| `POST`  | `/api/proposals` | `{ requestId, providerName, providerEmail, providerExpertise?, message, price }` | `201 { proposal }` (request → PROPOSED) |
| `PATCH` | `/api/proposals` | `{ proposalId, action: "accept" \| "reject" }`                           | `{ proposal }` (accept closes the request, rejects the rest) |
| `GET`   | `/api/status`    | `?requestId=` (optional)                                                  | `{ health, events }`                  |

Customers and providers are upserted by email on submit, so the forms are
self-contained (no separate sign-up step).

## Core flow

```
customer submits ServiceRequest
  -> AIBrief generated (deterministic fallback) + StatusEvent REQUEST_CREATED, BRIEF_GENERATED
  -> Provider submits a Proposal (request -> PROPOSED, StatusEvent PROPOSAL_SUBMITTED)
  -> Customer accepts/rejects (StatusEvent PROPOSAL_ACCEPTED/REJECTED; accept -> request ACCEPTED)
  -> customer / provider / admin dashboards render current state from the DB
```

## Testing without external services

- **Unit (Vitest).** `vitest.config.ts` sets `DATABASE_URL=file:./test.db` for
  the test workers. `tests/setup/global-setup.ts` deletes any previous
  `prisma/test.db` and runs a non-destructive `prisma db push` once to create
  the schema; `tests/setup/test-setup.ts` truncates all tables before each test.
  No external DB, no network. `fileParallelism: false` keeps the single SQLite
  file free of write contention. (We avoid `--force-reset` so the setup runs
  unattended in any CI — Prisma 6+ blocks that destructive flag under AI agents.)
- **E2E (Playwright).** `playwright.config.ts` has a `webServer` that runs
  `pnpm run e2e:server` — it resets and seeds a dedicated `prisma/e2e.db`, then
  starts `next dev` on port 3100. The smoke test drives the real UI through the
  full flow. Install the browser once with
  `pnpm exec playwright install chromium` (the browser cache is shared
  user-wide).

## Render Postgres profile (hosted)

The local profile is SQLite; the hosted profile is Postgres on Render. To switch
(do this in the deploy pipeline, not in this template):

1. In `prisma/schema.prisma`, change the datasource provider:

   ```prisma
   datasource db {
     provider = "postgresql"
     url      = env("DATABASE_URL")
   }
   ```

2. Set `DATABASE_URL` to the Render Postgres connection string, e.g.
   `postgresql://USER:PASSWORD@HOST:5432/DBNAME?sslmode=require`.

3. Generate a migration history for Postgres and apply it on deploy:

   ```bash
   prisma migrate dev --name init     # once, to create prisma/migrations
   prisma migrate deploy              # in the Render build/release step
   ```

4. A typical Render service runs `pnpm install && prisma generate && prisma
   migrate deploy && next build` as the build command and `next start` as the
   start command, with `DATABASE_URL` provided by a Render Postgres add-on.

Because every field type is portable, no schema changes are required beyond the
`provider` switch. A second datasource provider is **not** implemented in this
template — only documented — to keep the local profile simple. Generating the
`render.yaml` and running the deploy is the factory's deploy step (plan unit
U9), not part of this app.

## Handoff summary

- **What this is:** a working AI Services Marketplace (request → AI brief →
  proposal → accept/reject → status), persisted to a database, with customer,
  provider, and admin dashboards.
- **Profiles:** SQLite for local/dev/test (zero external services); Postgres on
  Render for hosting (provider switch documented above).
- **Determinism:** AI brief has a deterministic fallback, so unit tests and CI
  are reproducible with no API keys.
- **Gates passed locally:** install, `prisma generate` + `db push`, lint,
  typecheck, unit tests, `next build`, and a Playwright smoke test that boots the
  app and walks the full flow.
- **Deferred:** live AI provider integration (structured for, not implemented),
